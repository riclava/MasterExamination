# 二、必须掌握的知识点 + 练习题

## 1. 模块内聚类型

### 必须掌握

内聚表示：**一个模块内部各部分之间联系的紧密程度**。内聚越高，模块越独立，设计越好。

常见内聚从低到高大致为：

1. 偶然内聚：模块内部功能毫无关系。
2. 逻辑内聚：把逻辑上相似的功能放在一起，通过参数选择执行哪一个。
3. 时间内聚：一些操作因为在同一时间执行而放在一起。
4. 过程内聚：模块内操作按固定步骤执行，但数据联系不强。
5. 通信内聚：模块内多个操作使用同一输入数据或产生同一输出数据。
6. 顺序内聚：前一部分输出是后一部分输入。
7. 功能内聚：模块只完成一个明确功能，内聚最高。

### 本卷重点

题目中出现了：

> 模块内部各部分没什么联系，另一个模块通过传递基本参数向它传递一个控制。

这种情况属于 **逻辑内聚**。因为它是通过控制参数决定执行哪一类功能。

### 练习题

**题 1：**
某模块中包含“添加用户、删除用户、修改用户、查询用户”四个操作，通过传入 `opType` 参数决定执行哪一个操作。该模块属于哪种内聚？

答案：逻辑内聚。

**题 2：**
某模块负责“读取订单信息 → 计算订单金额 → 生成发票”，前一步的结果作为后一步的输入。该模块属于哪种内聚？

答案：顺序内聚。

**题 3：**
某模块中的多个功能都使用同一份客户资料，例如打印客户信息、统计客户年龄、检查客户等级，但这些功能之间没有明显先后关系。该模块属于哪种内聚？

答案：通信内聚。

## 2. DFD 数据流图与顶层图

### 必须掌握

DFD，Data Flow Diagram，数据流图，用来描述系统中的：

* 外部实体
* 处理过程
* 数据流
* 数据存储

DFD 顶层图也叫 **上下文图**，它把整个系统看成一个大处理过程，主要说明：

> 系统从外部接收什么输入，向外部输出什么结果。

### 本卷重点

DFD 顶层流图用于指出系统的 **输入与输出**，不是详细的数据存储，也不是内部处理过程。

### 练习题

**题 1：**
在学生成绩管理系统的 DFD 顶层图中，应该重点表现什么？

答案：学生、教师等外部实体与系统之间的输入和输出数据流。

**题 2：**
DFD 顶层图是否需要画出数据库表结构？

答案：不需要。顶层图重点表现系统边界、外部实体、输入输出。

**题 3：**
“用户提交订单，系统返回订单确认信息”，在 DFD 顶层图中，“提交订单”和“订单确认信息”分别属于什么？

答案：都是数据流。前者是输入数据流，后者是输出数据流。

## 3. 黑盒测试与白盒测试

### 必须掌握

黑盒测试：

* 不关心程序内部结构。
* 根据需求规格说明书测试功能。
* 重点是输入、输出、功能是否正确。
* 常用方法：等价类划分、边界值分析、因果图、判定表。

白盒测试：

* 关注程序内部逻辑。
* 重点测试语句、分支、路径、条件。
* 常用方法：语句覆盖、判定覆盖、条件覆盖、路径覆盖、基本路径测试。

### 本卷重点

黑盒测试主要考虑系统的 **功能**，不是程序流程、判定分支结构。

### 练习题

**题 1：**
登录功能要求用户名和密码正确才能登录。测试人员只根据需求设计“正确用户名+正确密码”“错误用户名”“错误密码”等测试用例，这属于什么测试？

答案：黑盒测试。

**题 2：**
测试人员根据代码中的 `if-else` 分支设计测试用例，确保每条分支都执行一次，这属于什么测试？

答案：白盒测试。

**题 3：**
输入年龄范围为 18～60，测试 17、18、60、61，这主要使用了哪种黑盒测试方法？

答案：边界值分析。

## 4. RUP 的核心概念

### 必须掌握

RUP，Rational Unified Process，统一软件开发过程，是一种迭代式、增量式的软件过程。

RUP 中有几个重要概念：

| 问题    | RUP 概念  |
| ----- | ------- |
| 谁做？   | 角色      |
| 做什么？  | 活动 / 任务 |
| 产生什么？ | 制品      |
| 怎么做？  | 工作流     |

### 本卷重点

题目问 RUP 中“谁做”对应什么概念，答案是 **角色**。

### 练习题

**题 1：**
RUP 中，系统分析员、设计员、测试员属于什么概念？

答案：角色。

**题 2：**
RUP 中，用例模型、设计模型、测试报告属于什么概念？

答案：制品。

**题 3：**
RUP 中描述开发过程中活动顺序和组织方式的是哪一类概念？

答案：工作流。

## 5. RUP 的 6 个核心工程工作流

### 必须掌握

本卷简答题直接考了 RUP 的 6 个核心工程工作流。

必须背熟：

1. 业务建模
   理解业务流程、业务目标和组织结构。

2. 需求
   收集、分析、定义系统的功能需求和非功能需求。

3. 分析与设计
   把需求转化为系统结构、类、接口、数据库、架构设计。

4. 实现
   编写代码，完成单元测试，把设计变成可运行软件。

5. 测试
   验证软件是否满足需求，发现并修复缺陷。

6. 部署
   将软件交付用户，包括安装、配置、培训、数据迁移等。

另外还应知道 RUP 的 3 个核心支持工作流：

1. 项目管理
2. 配置与变更管理
3. 环境

### 练习题

**题 1：**
RUP 中哪个工作流负责建立用例模型、明确功能需求和非功能需求？

答案：需求工作流。

**题 2：**
RUP 中哪个工作流负责把需求转化为类图、接口设计、数据库设计和架构设计？

答案：分析与设计工作流。

**题 3：**
RUP 中哪个工作流负责安装、配置、用户培训和软件交付？

答案：部署工作流。

## 6. 设计模式

### 必须掌握

设计模式是：

> 针对反复出现的软件设计问题，总结出来的可复用解决方案。

设计模式的核心价值：

* 复用优秀设计经验。
* 提高系统可维护性。
* 提高系统可扩展性。
* 降低设计复杂度。
* 与具体编程语言无关。

设计模式不是为了：

* 直接提高程序运行速度。
* 保证程序员一定更快写完代码。
* 只适用于某一种语言。

### 本卷重点

面向对象设计模式的正确说法是：**可以复用好的设计**。

### 练习题

**题 1：**
设计模式的核心作用是什么？

答案：复用成熟的设计经验，解决常见设计问题。

**题 2：**
“设计模式只能用于 Java，不能用于 Python 或 C++。”这句话对吗？

答案：错。设计模式是设计思想，不依赖特定语言。

**题 3：**
单例模式、工厂方法模式、观察者模式都属于什么？

答案：设计模式。

## 7. CMMI 过程改进模型

### 必须掌握

CMMI 是能力成熟度模型集成，用于软件过程改进。

它支持两种改进方式：

1. 阶段式表示
   关注组织整体成熟度等级。

2. 连续式表示
   关注单个过程域的能力等级。

### 本卷重点

题目说：

> CMMI 支持阶段性过程改进，不支持连续性过程改进。

这是错误的。CMMI 两种都支持。

### 练习题

**题 1：**
CMMI 是否只支持阶段式改进？

答案：不是。它同时支持阶段式和连续式改进。

**题 2：**
阶段式 CMMI 更关注什么？

答案：组织整体过程成熟度等级。

**题 3：**
连续式 CMMI 更关注什么？

答案：某个具体过程域的能力等级。

## 8. 软件过程模型：瀑布模型、迭代模型、敏捷模型

### 必须掌握

瀑布模型特点：

* 阶段线性推进。
* 需求分析、设计、编码、测试、维护依次进行。
* 适合需求明确、稳定、规模较小、变更较少的项目。

瀑布模型缺点：

* 不适合需求频繁变化。
* 后期修改成本高。
* 大型复杂项目风险较大。

大型系统通常更适合：

* 迭代模型
* 增量模型
* RUP
* 螺旋模型
* 敏捷开发

### 本卷重点

题目说：

> 大型系统，且具备相关领域知识和开发经验，此时采用瀑布模型是最好的选择。

这个说法错误。大型系统往往需求复杂、风险高、变化多，不一定适合瀑布模型。

### 练习题

**题 1：**
需求非常明确、变化很少的小型项目，适合采用哪种过程模型？

答案：瀑布模型。

**题 2：**
需求不稳定、用户需要不断反馈的大型项目，更适合采用哪类过程模型？

答案：迭代式、增量式、敏捷或螺旋模型。

**题 3：**
瀑布模型最大的风险是什么？

答案：需求变更适应能力差，后期修改成本高。

## 9. UML 关系：依赖、关联、聚合、组合、泛化、实现

### 必须掌握

UML 中常见关系：

| 关系 | 含义              | 例子              |
| -- | --------------- | --------------- |
| 依赖 | 一个类临时使用另一个类     | A 方法参数中使用 B     |
| 关联 | 两个类之间有结构性联系     | 学生选课            |
| 聚合 | 整体/部分，部分可独立存在   | 班级和学生           |
| 组合 | 强整体/部分，部分不能脱离整体 | 订单和订单明细         |
| 泛化 | 继承关系            | 学生是人            |
| 实现 | 类实现接口           | 类实现 Runnable 接口 |

### 本卷重点

整体/部分关系不能用依赖表示，应该用：

* 聚合
* 组合

所以“用依赖关系表示整体/部分关系”是错误的。

### 练习题

**题 1：**
汽车和发动机之间通常可以建模为什么 UML 关系？

答案：组合关系，尤其当发动机作为汽车组成部分不可独立于汽车生命周期考虑时。

**题 2：**
班级和学生之间通常可以建模为什么 UML 关系？

答案：聚合关系。学生可以脱离某个班级而存在。

**题 3：**
某个类的方法参数中使用了另一个类，但两者没有长期结构关系，这是什么关系？

答案：依赖关系。

## 10. 软件配置管理与质量控制

### 必须掌握

软件配置管理 SCM 的主要活动：

1. 配置项标识
2. 版本控制
3. 变更控制
4. 配置状态报告

通过测试发现缺陷、检查软件是否满足质量要求，属于什么活动？

答案：软件质量控制。

**题 3：**
配置审计属于质量控制还是配置管理？

答案：配置管理。

## 11. 软件风险管理

### 必须掌握

软件风险管理一般包括：

1. 风险识别
   找出可能发生的问题。

2. 风险预测 / 风险评估
   分析风险发生的可能性和影响程度。

3. 风险应对
   制定规避、转移、缓解、接受等措施。

4. 风险监控
   持续跟踪风险状态。

### 本卷重点

风险预测主要考虑两个方面：

1. 风险发生的可能性
2. 风险发生后的影响程度

不是“风险发生的可能性和风险消除方法”。

### 练习题

**题 1：**
“需求频繁变更可能导致项目延期”，这是哪类风险？

答案：需求风险 / 项目风险。

**题 2：**
风险预测主要分析哪两个因素？

答案：风险发生的概率和风险影响程度。

**题 3：**
“增加原型验证以减少需求不确定性”属于风险管理中的哪个阶段？

答案：风险应对，具体是风险缓解。

## 12. 开源软件

### 必须掌握

开源软件是指：

> 软件源代码可以被公众获取、查看、使用、修改和分发，并且通常遵守某种开源许可证。

常见开源许可证：

* GPL
* MIT
* Apache License
* BSD

典型例子：

* Linux
* MySQL
* PostgreSQL
* Python
* VS Code 的开源部分
* Kubernetes

注意：

开源不一定等于免费。
免费软件也不一定开源。
开源强调的是源代码开放和许可证允许的使用、修改、分发权利。

### 练习题

**题 1：**
什么是开源软件？请举一个例子。

答案：源代码可以被公众获取、查看、修改和分发的软件。例如 Linux。

**题 2：**
免费软件一定是开源软件吗？

答案：不一定。免费只说明不收费，不代表源代码开放。

**题 3：**
开源软件是否可以随意修改和分发，不受任何限制？

答案：不完全是。需要遵守对应开源许可证。

## 13. 结构化程序设计的三种基本控制结构

### 必须掌握

结构化程序设计要求复杂程序都用三种基本控制结构组合实现：

1. 顺序结构
2. 选择结构
3. 循环结构

每种结构都应该：

* 单入口
* 单出口
* 逻辑清晰
* 避免随意使用 goto

### 三种结构的类 C 语言表示

顺序结构：

```c
A();
B();
```

选择结构：

```c
if (condition) {
    A();
} else {
    B();
}
```

循环结构：

```c
while (condition) {
    A();
}
```

或者：

```c
do {
    A();
} while (condition);
```

### 本卷重点

简答题要求写出基本控制结构，并用类 C 语言或流程图描述。

### 练习题

**题 1：**
结构化程序设计的三种基本控制结构是什么？

答案：顺序结构、选择结构、循环结构。

**题 2：**
下面代码属于哪种结构？

```c
if (score >= 60) {
    printf("pass");
} else {
    printf("fail");
}
```

答案：选择结构。

**题 3：**
为什么结构化程序设计强调单入口、单出口？

答案：为了提高程序可读性、可测试性和可维护性，降低控制流复杂度。

## 14. UML 状态图

### 必须掌握

状态图用于描述对象或系统在不同事件驱动下的状态变化。

状态图中的核心元素：

1. 初始状态
2. 状态
3. 转移
4. 事件
5. 条件
6. 动作
7. 终止状态

常见表示：

```text
状态A --事件/条件--> 状态B
```

### 本卷重点

建模题给出“学生查询成绩”的业务流程，要求补全状态：

* 显示首页
* 要求学号
* 校验学号
* 显示错误信息
* 要求课程名
* 校验课程名
* 查询选课文件
* 显示成绩
* 询问用户操作
* 显示查询失败信息

做这种题时，要从题干中提取：

1. 用户动作
   例如：请求查询、输入学号、输入课程名、继续查询、结束查询。

2. 系统动作
   例如：显示首页、要求学号、查询学生文件、查询课程文件、查询选课文件。

3. 判断条件
   例如：学号正确/错误、课程名正确/错误、查询成功/失败。

4. 返回路径
   例如：错误后返回首页，继续查询后回到输入学号。

### 练习题

**题 1：**
在成绩查询系统中，用户输入学号后，系统检查学号是否存在。这个状态可以命名为什么？

答案：校验学号。

**题 2：**
用户输入错误课程名后，系统显示“课程名不正确”并返回首页。这里至少涉及哪些状态？

答案：校验课程名、显示错误信息、显示首页。

**题 3：**
查询成功后，用户可以选择继续查询或结束查询，这个地方在状态图中通常对应什么？

答案：一个分支转移，或者设置“询问用户操作”状态，再根据用户选择转移到“要求学号”或“首页”。

## 15. McCabe 环路复杂度与基本路径覆盖

### 必须掌握

McCabe 环路复杂度用于度量程序控制流复杂度。

常用计算方法：

### 方法一：判定节点数 + 1

```text
V(G) = 判定节点数 + 1
```

判定节点包括：

* if
* while
* for
* switch 分支
* 条件判断
* 状态图中的分支判断

### 方法二：边数 - 节点数 + 2

```text
V(G) = E - N + 2
```

如果有多个连通区域：

```text
V(G) = E - N + 2P
```

其中 P 是连通区域数。

### 本卷重点

建模题中有 4 个判断：

1. 学号是否正确
2. 课程名是否正确
3. 查询是否成功
4. 用户选择继续查询还是结束查询

所以：

```text
V(G) = 4 + 1 = 5
```

因此至少需要 5 条基本路径。

### 基本路径覆盖答题模板

可以这样写：

1. 学号错误路径
   首页 → 输入学号 → 校验学号 → 显示学号错误 → 首页

2. 课程名错误路径
   首页 → 输入学号 → 校验学号正确 → 输入课程名 → 校验课程名错误 → 首页

3. 查询失败路径
   首页 → 输入学号 → 学号正确 → 输入课程名 → 课程名正确 → 查询选课文件 → 查询失败 → 首页

4. 查询成功后结束路径
   首页 → 输入学号 → 学号正确 → 输入课程名 → 课程名正确 → 查询成功 → 显示成绩 → 结束查询 → 首页

5. 查询成功后继续路径
   首页 → 输入学号 → 学号正确 → 输入课程名 → 课程名正确 → 查询成功 → 显示成绩 → 继续查询 → 输入学号

### 练习题

**题 1：**
某程序有 3 个 `if` 判断，没有循环，McCabe 环路复杂度是多少？

答案：3 + 1 = 4。

**题 2：**
某流程中有“用户名是否正确”“密码是否正确”“验证码是否正确”三个判断，至少需要几条基本路径？

答案：4 条。

**题 3：**
基本路径覆盖的目的是什么？

答案：设计一组测试用例，使程序中的每条独立路径至少执行一次。

# 三、按题型复习策略

## 选择题

重点背概念区分：

* 逻辑内聚 vs 通信内聚 vs 顺序内聚
* DFD 顶层图的作用
* 黑盒测试关注功能
* RUP 中角色、制品、工作流的区别
* 设计模式的价值是复用设计

## 判断题

重点背容易混淆的“是/不是”：

* CMMI 同时支持阶段式和连续式改进。
* 瀑布模型不适合需求变化大的大型系统。
* UML 整体/部分关系用聚合或组合，不用依赖。
* 质量控制不是配置管理活动。
* 风险预测看概率和影响，不是看消除方法。

## 简答题

必须能完整写出：

1. 开源软件定义 + 例子。
2. RUP 6 个核心工程工作流及目的。
3. 结构化程序设计三种基本控制结构。

## 建模题

必须会：

1. 从题干抽取状态。
2. 从题干抽取事件和条件。
3. 补全状态图中的状态名称。
4. 找判断节点。
5. 计算 McCabe 环路复杂度。
6. 写出基本路径覆盖测试路径。

# 四、最小背诵清单

考前至少背熟下面这些：

1. 内聚类型：逻辑内聚通过控制参数选择功能。
2. DFD 顶层图：表示系统与外部实体之间的输入输出。
3. 黑盒测试：按需求测功能，不看内部结构。
4. RUP：谁做 = 角色，产物 = 制品，怎么做 = 工作流。
5. 设计模式：复用好的设计。
6. CMMI：支持阶段式和连续式改进。
7. 瀑布模型：适合需求稳定，不适合大型复杂易变系统。
8. UML 整体/部分：聚合或组合，不是依赖。
9. 配置管理：版本、变更、配置项、审计。
10. 质量控制：测试、评审、发现缺陷。
11. 风险预测：概率 + 影响。
12. 开源软件：源代码开放，可按许可证使用、修改、分发。
13. 结构化程序三结构：顺序、选择、循环。
14. McCabe：判定节点数 + 1。
15. 基本路径覆盖：复杂度是多少，至少设计多少条独立路径。
