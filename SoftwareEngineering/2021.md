# 二、知识点 1：面向对象基础

## 必须掌握

面向对象考试最常考的是**概念辨析**。

### 1. 封装

把对象的数据和操作数据的方法绑定在一起，对外隐藏内部实现细节。

关键词：

> 信息隐藏、数据保护、接口访问

### 2. 继承

子类获得父类的属性和方法。

关键词：

> 代码复用、父类、子类、泛化、特化

### 3. 多态

不同类的对象对同一消息作出不同响应。

例如：

```text
Animal.speak()
Dog.speak() -> 汪汪
Cat.speak() -> 喵喵
```

看到题干中出现：

> 同一消息、不同对象、不同响应

基本就是考 **多态**。

### 4. 重载

同一个类中，方法名相同，但参数列表不同。

例如：

```java
print(int x)
print(String s)
print(int x, int y)
```

关键词：

> 同名方法、参数不同、编译期决定

### 5. 重写

子类重新定义父类已有的方法。

关键词：

> 子类覆盖父类方法、运行时动态绑定、多态基础

### 6. 对象生命周期

对象不是永远处于活动状态。

对象可能处于：

```text
创建 -> 使用/等待 -> 不再引用 -> 回收
```

所以“程序运行时所有已创建对象都始终处于活动状态”是错误的。

---

## 练习题

### 练习 1

在面向对象程序设计中，同一操作作用于不同对象，可以产生不同执行结果，这种机制称为：

A. 封装
B. 继承
C. 多态
D. 抽象

答案：C

---

### 练习 2

下列关于方法重载的说法，正确的是：

A. 重载一定发生在父类和子类之间
B. 重载要求方法名相同、参数列表不同
C. 重载是运行时动态绑定
D. 重载必须返回值类型不同

答案：B

---

### 练习 3

判断：对象一旦被创建，在程序结束前一定始终处于活动状态。

答案：×
解析：对象可能处于等待状态、不可达状态，也可能被垃圾回收。

# 三、知识点 2：软件生命周期阶段任务

## 必须掌握

软件工程题很爱考：

> 某项工作属于哪个阶段？

你必须能快速区分下面几个阶段。

## 1. 需求分析阶段

核心问题：

> 系统应该做什么？

主要任务：

```text
理解用户需求
定义功能需求
定义非功能需求
建立系统逻辑模型
编写需求规格说明书 SRS
```

常见模型：

```text
用例图
数据流图 DFD
实体关系图 ER
状态图
业务流程图
```

注意：需求分析不是设计“怎么实现”，而是说明“要实现什么”。

---

## 2. 概要设计阶段

核心问题：

> 系统总体怎么分？

主要任务：

```text
确定系统总体结构
划分模块
定义模块之间关系
定义模块接口
确定体系结构
```

试卷中单选题考的就是这个点：概要设计主要确定**软件系统总体结构和模块间关系**。

---

## 3. 详细设计阶段

核心问题：

> 每个模块内部怎么实现？

主要任务：

```text
设计模块内部算法
设计数据结构
设计详细流程
编写详细设计说明书
```

---

## 4. 编码阶段

核心问题：

> 用程序语言实现设计结果。

主要任务：

```text
编写代码
遵守编码规范
命名规范
注释规范
代码审查
```

---

## 5. 测试阶段

核心问题：

> 系统是否满足需求、是否存在缺陷？

主要任务：

```text
单元测试
集成测试
系统测试
验收测试
```

---

## 6. 维护阶段

核心问题：

> 软件交付后如何修改、完善、适应环境变化？

主要任务：

```text
修复错误
适应环境
改进性能
增强功能
提高可维护性
```

---

## 练习题

### 练习 1

软件需求分析阶段的主要任务是：

A. 确定模块之间的调用关系
B. 编写源代码
C. 明确用户需求并建立系统逻辑模型
D. 设计模块内部算法

答案：C

---

### 练习 2

软件概要设计阶段主要解决的问题是：

A. 每个函数的具体代码如何写
B. 系统总体结构和模块关系如何安排
C. 测试用例如何设计
D. 用户手册如何编写

答案：B

---

### 练习 3

判断：详细设计阶段主要确定软件系统的总体结构和模块之间关系。

答案：×
解析：这是概要设计的任务。详细设计主要确定模块内部算法、数据结构和处理流程。

# 四、知识点 3：需求分析

## 必须掌握

需求分析是考试高频点，重点不是背定义，而是理解它和设计的区别。

## 需求分析的本质

需求分析要回答：

```text
用户需要什么？
系统应该提供哪些功能？
系统有哪些性能、安全、可靠性等约束？
系统和外部环境如何交互？
```

## 需求类型

### 1. 功能需求

描述系统必须提供的功能。

例如：

```text
用户可以登录
用户可以购票
系统可以计算票价
系统可以生成订单
```

### 2. 非功能需求

描述系统质量要求。

例如：

```text
响应时间不超过 2 秒
系统可用性达到 99.9%
密码必须加密存储
支持 1000 人并发访问
```

### 3. 约束条件

描述开发或运行限制。

例如：

```text
必须使用 Java 开发
必须部署在 Linux 服务器
必须兼容某浏览器
```

## 需求分析产物

常见产物：

```text
需求规格说明书 SRS
用例模型
数据流图
数据字典
ER 图
状态图
业务流程图
```

---

## 练习题

### 练习 1

下面哪一项属于非功能需求？

A. 用户可以修改密码
B. 管理员可以删除用户
C. 系统响应时间不超过 1 秒
D. 用户可以查询订单

答案：C

---

### 练习 2

判断：需求分析阶段主要研究软件系统应该“如何实现”。

答案：×
解析：需求分析主要研究“系统做什么”，设计阶段才研究“如何实现”。

---

### 练习 3

某图书管理系统要求：“读者可以查询图书、借阅图书、归还图书；系统应在 3 秒内返回查询结果。”
请指出其中的功能需求和非功能需求。

参考答案：

```text
功能需求：查询图书、借阅图书、归还图书。
非功能需求：查询结果应在 3 秒内返回。
```

# 五、知识点 4：软件设计：概要设计与详细设计

## 必须掌握

软件设计分为：

```text
概要设计
详细设计
```

## 概要设计

也叫总体设计、架构设计。

重点：

```text
系统分成哪些模块？
模块之间如何调用？
模块之间传递什么数据？
整体架构是什么？
```

常见内容：

```text
模块划分
模块层次结构
模块接口
数据库总体结构
系统架构
```

## 详细设计

重点：

```text
每个模块内部怎么做？
算法是什么？
数据结构是什么？
控制流程是什么？
```

常见内容：

```text
流程图
伪代码
判定表
判定树
算法描述
数据结构设计
```

## 高频区分

| 内容     | 所属阶段                 |
| ------ | -------------------- |
| 系统总体结构 | 概要设计                 |
| 模块划分   | 概要设计                 |
| 模块间接口  | 概要设计                 |
| 具体算法   | 详细设计                 |
| 具体数据结构 | 详细设计                 |
| 代码命名规范 | 编码阶段                 |
| 界面按钮布局 | 界面设计/详细设计相关，不是概要设计核心 |

---

## 练习题

### 练习 1

下列属于详细设计任务的是：

A. 确定系统模块之间关系
B. 设计模块内部算法和数据结构
C. 确定软件总体架构
D. 明确用户需求

答案：B

---

### 练习 2

判断：概要设计的重点是确定系统总体结构，而不是每个模块的具体算法。

答案：√

---

某系统被划分为用户管理、订单管理、支付管理、库存管理四个模块，并定义了模块间接口。该工作属于哪个阶段？

# 六、知识点 5：内聚与耦合

## 必须掌握

软件设计的核心原则：

```text
高内聚，低耦合
```

## 1. 内聚

内聚表示：

> 一个模块内部各元素之间联系的紧密程度。

目标：

```text
内聚越高越好
```

## 2. 耦合

耦合表示：

> 不同模块之间依赖关系的紧密程度。

目标：

```text
耦合越低越好
```

所以：

> “模块内聚性越高越好，耦合性越高越好” 是错误的。

正确说法是：

> 内聚性越高越好，耦合性越低越好。

---

## 内聚类型排序

从低到高一般为：

```text
偶然内聚
< 逻辑内聚
< 时间内聚
< 过程内聚
< 通信内聚
< 顺序内聚
< 功能内聚
```

## 试卷重点

试卷选项里出现：

```text
逻辑内聚
过程内聚
通信内聚
顺序内聚
```

其中最高的是：

```text
顺序内聚
```

因为功能内聚没有出现在选项中。

---

## 各类内聚理解

### 1. 偶然内聚

模块内各功能没有明显关系。

例如：

```text
一个模块里既打印报表，又计算工资，又清理缓存。
```

最差。

---

### 2. 逻辑内聚

模块内功能逻辑上类似，但通过参数选择执行哪个。

例如：

```text
输入处理模块：
flag = 1 处理键盘输入
flag = 2 处理文件输入
flag = 3 处理网络输入
```

---

### 3. 时间内聚

模块内操作因为在同一时间执行而放在一起。

例如：

```text
系统初始化模块：
打开日志
加载配置
连接数据库
初始化缓存
```

---

### 4. 过程内聚

模块内操作按某个过程顺序执行，但数据关系不强。

---

### 5. 通信内聚

模块内操作处理同一数据集合。

例如：

```text
对同一批学生成绩：
计算平均分
计算最高分
计算最低分
```

---

### 6. 顺序内聚

前一个处理的输出是后一个处理的输入。

例如：

```text
读取订单 -> 校验订单 -> 计算金额 -> 生成支付请求
```

---

### 7. 功能内聚

模块只完成一个明确功能。

例如：

```text
calculateTotalPrice()
validatePassword()
sendVerificationCode()
```

最高。

---

## 练习题

### 练习 1

下列内聚类型中，内聚度最高的是：

A. 逻辑内聚
B. 过程内聚
C. 通信内聚
D. 顺序内聚

答案：D

---

### 练习 2

判断：优秀的软件设计应追求高内聚、低耦合。

答案：√

---

### 练习 3

某模块中的处理步骤为：读取订单、校验订单、计算订单金额、生成支付信息，并且每一步输出作为下一步输入。该模块属于哪种内聚？

答案：顺序内聚。

# 七、知识点 6：黑盒测试与白盒测试

## 必须掌握

## 1. 黑盒测试

黑盒测试不关心程序内部结构，只关心：

```text
输入是什么？
输出是否正确？
功能是否符合需求？
```

关键词：

```text
功能
需求规格说明
输入输出
不看代码
```

常见方法：

```text
等价类划分
边界值分析
因果图
判定表
错误推测法
场景法
```

试卷中考点：

> 黑盒测试主要关注功能是否符合需求规格说明。

---

## 2. 白盒测试

白盒测试关注程序内部结构。

关键词：

```text
代码
路径
语句
分支
条件
循环
逻辑结构
```

常见覆盖准则：

```text
语句覆盖
判定覆盖
条件覆盖
判定-条件覆盖
条件组合覆盖
路径覆盖
```

---

## 黑盒 vs 白盒

| 对比项   | 黑盒测试        | 白盒测试           |
| ----- | ----------- | -------------- |
| 是否看代码 | 不看          | 看              |
| 依据    | 需求规格说明      | 程序内部结构         |
| 关注点   | 功能正确性       | 逻辑路径、分支、条件     |
| 常见方法  | 等价类、边界值、判定表 | 语句覆盖、分支覆盖、路径覆盖 |
| 适合人员  | 测试人员、用户代表   | 开发人员、测试人员      |

---

## 练习题

### 练习 1

黑盒测试主要依据：

A. 程序源代码
B. 控制流程图
C. 需求规格说明书
D. 程序内部路径

答案：C

---

### 练习 2

下列属于白盒测试关注内容的是：

A. 用户界面是否美观
B. 功能是否符合需求
C. 判定条件和分支覆盖情况
D. 用户是否满意

答案：C

---

### 练习 3

判断：黑盒测试不关心程序内部实现细节。

答案：√

# 八、知识点 7：测试阶段分类

## 必须掌握

软件测试按阶段通常分为：

```text
单元测试
集成测试
系统测试
验收测试
```

---

## 1. 单元测试

测试对象：

```text
单个模块、函数、类、组件
```

主要目标：

```text
发现模块内部逻辑错误
```

不是主要发现模块接口之间错误。

所以试卷判断题：

> 单元测试的主要目的是发现模块接口之间的错误。

答案是：

```text
×
```

因为模块接口错误主要是集成测试发现。

---

## 2. 集成测试

测试对象：

```text
多个模块组合之后的系统部分
```

主要目标：

```text
发现模块之间接口错误
模块协作错误
数据传递错误
调用关系错误
```

---

## 3. 系统测试

测试对象：

```text
完整软件系统
```

主要目标：

```text
验证整个系统是否满足需求规格说明
```

---

## 4. 验收测试

测试主体通常是：

```text
用户或客户
```

主要目标：

```text
判断软件是否可以交付使用
```

---

### 练习 3

# 九、知识点 8：测试覆盖准则与最少测试用例

## 必须掌握

试卷简答题考了：

> 给一段程序，设计最少测试用例满足语句覆盖。

所以你必须掌握几种覆盖准则。

---

## 1. 语句覆盖

要求：

```text
程序中每条可执行语句至少执行一次。
```

特点：

```text
覆盖要求最低
测试用例数量通常最少
但发现错误能力较弱
```

---

## 2. 判定覆盖 / 分支覆盖

要求：

```text
每个判断的真分支和假分支都至少执行一次。
```

例如：

```c
if (x > 0)
```

需要：

```text
x > 0 为真一次
x > 0 为假一次
```

---

## 3. 条件覆盖

要求：

```text
每个判断中每个条件的真假都至少出现一次。
```

例如：

```c
if (a > 0 && b > 0)
```

需要：

```text
a > 0 真/假都出现
b > 0 真/假都出现
```

---

## 4. 路径覆盖

要求：

```text
程序中所有可能路径至少执行一次。
```

最强，但测试用例数量可能爆炸。

---

## 试卷例题思想

程序：

```text
min = a
if (b < min) min = b
if (c < min) min = c
输出 min
```

要做到语句覆盖，只要让两个赋值语句都执行即可：

```text
a = 3, b = 2, c = 1
```

执行过程：

```text
min = 3
b < min 成立，min = 2
c < min 成立，min = 1
输出 1
```

所以最少 1 个测试用例即可。

---

## 练习题

### 练习 1

下面程序要满足语句覆盖，最少需要几个测试用例？

```c
if (x > 0) {
    y = 1;
}
print(y);
```

参考答案：1 个。
例如：

```text
x = 1
```

因为可以执行 if 内部语句和 print 语句。

---

### 练习 2

下面程序要满足判定覆盖，至少需要几个测试用例？

```c
if (x > 0) {
    y = 1;
} else {
    y = -1;
}
```

参考答案：2 个。

```text
x = 1，覆盖真分支
x = 0，覆盖假分支
```

---

### 练习 3

为下面程序设计一个满足语句覆盖的测试用例：

```c
min = a;
if (b < min) min = b;
if (c < min) min = c;
print(min);
```

参考答案：

```text
输入：a = 5, b = 3, c = 1
预期输出：1
```

# 十、知识点 9：软件维护

## 必须掌握

软件维护不是只“完善功能”，也包括纠错。

软件维护通常分为四类：

```text
纠错性维护
适应性维护
完善性维护
预防性维护
```

---

## 1. 纠错性维护

修复交付后发现的错误。

例如：

```text
修复登录失败 bug
修复支付金额计算错误
修复系统崩溃问题
```

---

## 2. 适应性维护

为了适应外部环境变化。

例如：

```text
操作系统升级
数据库版本升级
浏览器兼容性变化
第三方支付接口变更
```

---

## 3. 完善性维护

为了增强功能、改善性能或提升用户体验。

例如：

```text
增加批量导入功能
优化查询速度
改善界面交互
```

---

## 4. 预防性维护

为了提高未来可维护性和可靠性。

例如：

```text
重构代码
补充单元测试
优化模块结构
消除潜在风险
```

---

## 练习题

### 练习 1

软件上线后发现订单金额计算错误，开发人员进行修复，这属于：

B. 纠错性维护
C. 完善性维护
D. 预防性维护

答案：B

---

### 练习 2

A. 适应性维护
B. 纠错性维护

D. 逆向维护

---

### 练习 3

答案：×

# 十四、知识点 13：UML 状态图

## 必须掌握

状态图是本试卷最大分值题，必须重点掌握。

状态图用于描述：

> 一个对象或系统在不同事件触发下，从一个状态转换到另一个状态的过程。

---

## 状态图基本元素

### 1. 状态

表示系统在某一时刻的情况。

例如：

```text
显示首页
请输入起点站
验证起点站
请输入终点站
确认购票信息
验证支付结果
购票成功
支付失败
```

---

### 2. 转移

表示从一个状态到另一个状态。

例如：

```text
显示首页 -> 输入起点站
```

---

### 3. 事件

触发状态变化的动作。

例如：

```text
点击购票
输入起点站
输入终点站
确认支付
取消购票
```

---

### 4. 条件 / 守卫条件

判断转移是否发生。

例如：

```text
起点站有效
起点站无效
终点站有效
支付成功
支付失败
```

---

### 5. 动作

状态进入或转换时执行的操作。

例如：

```text
do: 显示首页
do: 显示错误信息
do: 计算票价
do: 出票
```

---

## 状态图解题方法

拿到文字描述后，按下面步骤做。

### 第一步：找状态

看系统处于哪些“等待、显示、验证、确认、处理”阶段。

例如试卷中的自动售票系统：

```text
首页
输入起点站
验证起点站
显示起点站错误
输入终点站
验证终点站
显示终点站错误
确认购票信息
验证支付结果
支付失败
购票成功
```

---

### 第二步：找事件

找用户或系统触发的动作。

```text
点击购票
输入起点站
输入终点站
确认支付
取消购票
返回首页
```

---

### 第三步：找判断条件

```text
起点站有效？
终点站有效且不同于起点？
支付成功？
```

---

### 第四步：找异常路径

考试很喜欢考异常路径：

```text
起点站无效 -> 显示错误 -> 返回首页
终点站无效 -> 显示错误 -> 返回首页
取消购票 -> 返回首页
支付失败 -> 显示失败 -> 返回首页
```

---

### 第五步：补全状态名

根据箭头前后的事件和条件判断状态名称。

例如：

```text
输入起点站之后，下一个状态通常是验证起点站。
验证成功之后，进入输入终点站。
验证失败之后，进入显示起点站不存在。
```

---

## 试卷状态图对应关系

根据题目描述，状态可整理为：

| 状态  | 含义            |
| --- | ------------- |
| S0  | 显示首页          |
| S1  | 购票 / 请输入起点站   |
| S2  | 验证起点站是否有效     |
| S3  | 显示起点站不存在      |
| S4  | 验证终点站是否有效     |
| S5  | 显示终点站无效或与起点相同 |
| S6  | 确认购票信息        |
| S7  | 显示取消购票信息      |
| S8  | 验证支付是否成功      |
| S9  | 显示支付失败        |
| S10 | 显示购票成功 / 出票成功 |

注意：考试图中可能只要求补某几个状态名称，核心是理解流程，不要死背编号。

---

# 十五、按题型给你的满分策略

## 1. 单项选择题

重点考概念辨析。

必须熟记：

```text
多态：同一消息，不同对象，不同响应
概要设计：总体结构、模块关系
黑盒测试：功能是否符合需求
最高内聚：功能内聚最高；若无功能内聚，则选顺序内聚
需求分析：定义用户需求，建立逻辑模型
```

---

## 2. 判断题

重点考“绝对化说法”。

看到这些表达要警惕：

```text
一定
始终
只包括
不包括
越高越好
全部
必须
```

例如：

```text
所有对象始终活动：错
单元测试主要发现接口错误：错
维护不包括纠错：错
耦合越高越好：错
CMMI 支持连续式和阶段式：对
```

---

## 3. 简答题

答题要有结构，建议用：

```text
概念 + 特点 + 示例
```

或者：

```text
名称 + 主要活动 + 说明
```

例如 RAD：

```text
RAD 包括业务建模、数据建模和流程建模。
业务建模分析业务信息流；
数据建模识别数据对象、属性和关系；
流程建模描述数据对象如何被增加、修改、删除和查询，以实现业务功能。
```

---

## 4. 建模题

建模题不要乱画，先做四步：

```text
找状态
找事件
找条件
找返回路径
```

尤其注意：

```text
错误路径
取消路径
支付失败路径
返回首页路径
```

# 十六、最终复习清单

考前你至少要能脱口而出下面内容：

```text
1. 多态：同一消息，不同对象，不同响应。
2. 重载：同名方法，参数不同。
3. 需求分析：定义用户需求，建立系统逻辑模型。
4. 概要设计：确定系统总体结构和模块关系。
5. 详细设计：确定模块内部算法和数据结构。
6. 黑盒测试：依据需求规格说明，关注功能。
7. 白盒测试：依据代码结构，关注路径、分支、条件。
8. 单元测试：测模块内部。
9. 集成测试：测模块接口和协作。
10. 高内聚、低耦合。
11. 内聚排序：偶然 < 逻辑 < 时间 < 过程 < 通信 < 顺序 < 功能。
12. 软件维护包括：纠错性、适应性、完善性、预防性。
13. CMMI 支持阶段式和连续式过程改进。
14. RAD 三种建模：业务建模、数据建模、流程建模。
15. Web 应用：B/S 架构，浏览器访问，HTTP/HTTPS，服务器端集中部署。
16. 状态图：状态、事件、转移、条件、动作。
```

重点优先级建议：

```text
第一优先级：软件生命周期、测试、内聚耦合、UML 状态图
第二优先级：面向对象、需求分析、维护
第三优先级：CMMI、RAD、Web 应用概念
```

这套试卷难度不大，真正要拿满分，关键不是背很多内容，而是把**概念边界**分清楚：
**需求 vs 设计，概要设计 vs 详细设计，黑盒 vs 白盒，单元测试 vs 集成测试，高内聚 vs 低耦合，多态 vs 重载。**
