# 软件工程满分知识点与练习题汇总

## 1. 软件生命周期阶段

### 必须掌握

软件生命周期通常包括：

```text
可行性研究 → 需求分析 → 概要设计 → 详细设计 → 编码 → 测试 → 运行维护
```

其中考试最常考的是：

| 阶段          | 核心问题       | 主要任务                     | 主要产物                    |
| ----------- | ---------- | ------------------------ | ----------------------- |
| 需求分析        | 软件要做什么     | 获取、分析、定义需求，建立系统模型        | 需求规格说明书、需求模型            |
| 概要设计 / 总体设计 | 系统整体怎么组织   | 划分模块，确定软件结构、模块接口、数据库总体结构 | 软件结构图、模块结构图             |
| 详细设计        | 每个模块内部怎么实现 | 设计算法、数据结构、接口细节、控制流程      | 详细设计说明书、流程图、N-S 图、PAD 图 |
| 编码          | 用程序实现设计    | 编写源代码                    | 程序代码                    |
| 测试          | 软件是否正确     | 发现错误、验证需求是否满足            | 测试报告、缺陷报告               |
| 维护          | 交付后如何修改完善  | 改正错误、适应环境、增强功能           | 修改后的软件版本                |

### 常见考法

这类题通常会问：

* 需求分析阶段的任务是什么？
* 总体设计结束后得到什么？
* 模块算法属于哪个阶段？
* 编码阶段的产物是什么？
* 测试计划在哪个阶段制定？

### 答题模板

如果问“某阶段的主要任务”，可以这样答：

```text
某阶段的主要任务是……，它关注……，主要产物是……。
```

例如：

```text
需求分析阶段的主要任务是定义软件需求并建立系统模型，它关注软件“做什么”，主要产物是需求规格说明书和需求模型。
```

### 易错点

| 错误理解           | 正确理解             |
| -------------- | ---------------- |
| 需求分析阶段给出软件处理方案 | 错，处理方案属于设计阶段     |
| 总体设计给出模块算法     | 错，模块算法属于详细设计     |
| 编码阶段设计模块结构     | 错，模块结构属于概要设计     |
| 测试阶段才制定测试计划    | 错，测试计划应在需求分析阶段制定 |

### 练习题

#### 题 1

软件开发中，需求分析阶段的核心任务是什么？

A. 给出系统模块结构
B. 定义模块算法
C. 定义需求并建立系统模型
D. 编写程序代码

**答案：C**

解析：需求分析阶段解决的是“软件要做什么”，不是“怎么实现”。

---

#### 题 2

“确定系统模块划分、模块之间的接口和软件总体结构”属于哪个阶段？

A. 需求分析
B. 概要设计
C. 详细设计
D. 编码

**答案：B**

解析：概要设计又叫总体设计，重点是系统结构和模块划分。

---

#### 题 3

“定义模块内部算法和处理流程”属于哪个阶段？

A. 需求分析
B. 概要设计
C. 详细设计
D. 测试

**答案：C**

解析：模块内部算法是详细设计的任务。

## 2. 需求分析与测试计划

### 必须掌握

需求分析阶段不仅要明确系统功能，还要为后续测试提供依据。

测试计划一般在**需求分析阶段**制定。

原因是：

1. 测试要验证软件是否满足需求；
2. 测试对象、测试范围、测试目标都来自需求；
3. 早期制定测试计划可以避免后期遗漏重要功能；
4. 测试阶段主要是执行测试，而不是才开始计划测试。

### 测试计划通常包括

| 内容   | 说明              |
| ---- | --------------- |
| 测试目标 | 要验证哪些功能和质量属性    |
| 测试范围 | 哪些模块、功能需要测试     |
| 测试策略 | 使用黑盒、白盒、集成测试等方法 |
| 测试资源 | 人员、环境、工具        |
| 测试进度 | 测试时间安排          |
| 通过标准 | 软件达到什么标准算通过     |

### 常见考法

判断题、选择题常考：

```text
确定测试计划是在测试阶段制定的。
```

这个说法是错误的。

正确说法：

```text
测试计划一般在需求分析阶段制定。
```

### 答题模板

```text
测试计划应在需求分析阶段制定。因为测试的目标是验证软件是否满足需求，测试内容、测试范围和测试标准都应以需求规格说明为依据。
```

### 易错点

| 说法              | 正误 |
| --------------- | -- |
| 测试计划在测试阶段制定     | 错  |
| 测试计划依据需求制定      | 对  |
| 需求分析只关注功能，不影响测试 | 错  |
| 测试阶段主要执行测试计划    | 对  |

### 练习题

#### 题 1

软件测试计划通常在哪个阶段制定？

A. 需求分析
B. 详细设计
C. 编码
D. 测试

**答案：A**

---

#### 题 2

为什么测试计划应在需求分析阶段制定？

**参考答案：**

因为测试的目标是验证软件是否满足需求，测试范围、测试内容和测试通过标准都依赖需求规格说明，所以应在需求明确后尽早制定测试计划。

---

#### 题 3

判断题：测试阶段的主要任务是制定测试计划。

**答案：×**

解析：测试阶段主要任务是执行测试计划、发现并记录错误。

---

## 3. 软件过程与基本过程

### 必须掌握

按软件过程的活动主体和工作内容分类，软件过程通常包括：

```text
基本过程、支持过程、组织过程
```

其中基本过程包括：

| 基本过程 | 含义                 |
| ---- | ------------------ |
| 获取过程 | 需方获取软件产品或服务的过程     |
| 供应过程 | 供方提供软件产品或服务的过程     |
| 开发过程 | 软件需求、设计、编码、测试等开发活动 |
| 运行过程 | 软件在实际环境中的运行活动      |
| 维护过程 | 软件交付后的修改、完善和适应活动   |

### 常见考法

试卷中考过：

```text
按不同人员的工作内容来分类软件过程，基本过程包括什么？
```

正确方向是：

```text
获取过程、供应过程、开发过程、运行过程、维护过程。
```

### 易错点

| 过程   | 是否属于基本过程          |
| ---- | ----------------- |
| 获取过程 | 是                 |
| 供应过程 | 是                 |
| 开发过程 | 是                 |
| 运行过程 | 是                 |
| 维护过程 | 是                 |
| 改善过程 | 否，通常属于组织过程        |
| 培训过程 | 否，通常属于组织过程或支持相关过程 |
| 配置管理 | 否，通常属于支持过程        |

### 练习题

#### 题 1

软件过程中的基本过程包括哪些？

**参考答案：**

包括获取过程、供应过程、开发过程、运行过程和维护过程。

---

#### 题 2

判断题：改善过程属于软件基本过程。

**答案：×**

解析：改善过程通常属于组织过程，不属于基本过程。

---

#### 题 3

下列属于软件基本过程的是：

A. 改善过程
B. 获取过程
C. 培训过程
D. 质量保证过程

**答案：B**

## 4. 结构化方法

### 必须掌握

结构化方法是软件工程中的经典方法，主要包括：

```text
结构化分析、结构化设计、结构化程序设计
```

其总的指导原则是：

```text
自顶向下，逐步求精
```

### 三个层次

| 方法      | 英文缩写 | 关注点       | 常用工具            |
| ------- | ---- | --------- | --------------- |
| 结构化分析   | SA   | 需求、数据流、加工 | DFD、数据字典        |
| 结构化设计   | SD   | 模块划分、模块结构 | 模块结构图           |
| 结构化程序设计 | SP   | 程序控制结构    | N-S 图、PAD 图、流程图 |

### 自顶向下、逐步求精

含义：

1. 先看整体系统；
2. 再分解成子系统；
3. 再分解成模块；
4. 最后细化到具体处理逻辑。

例如：

```text
图书管理系统
→ 借书处理、还书处理、购书处理、库存查询
→ 借书处理再细分为查询库存、登记借阅、发出通知
```

### 常见考法

判断题：

```text
在软件开发方法中，结构化方法总的指导原则是自顶向下、逐步求精。
```

答案是：

```text
√
```

### 易错点

| 错误说法         | 正确说法           |
| ------------ | -------------- |
| 结构化方法以对象为中心  | 错，面向对象方法以对象为中心 |
| 结构化分析以数据流为中心 | 对              |
| 结构化方法不使用抽象   | 错，也使用抽象        |
| 结构化方法强调自底向上  | 错，强调自顶向下       |

### 练习题

#### 题 1

结构化方法的总指导原则是什么？

**参考答案：**

自顶向下，逐步求精。

---

#### 题 2

结构化分析方法通常使用哪些工具？

**参考答案：**

常用数据流图 DFD 和数据字典。

---

#### 题 3

判断题：结构化方法强调从程序代码开始，逐步组合成系统。

**答案：×**

解析：结构化方法强调自顶向下，而不是从代码自底向上组合。

---

## 5. 面向对象分析与结构化分析的区别

### 必须掌握

面向对象分析和结构化分析的核心区别在于**分析视角不同**。

| 对比项  | 面向对象分析        | 结构化分析            |
| ---- | ------------- | ---------------- |
| 出发点  | 问题域中的对象、类、实体  | 数据流、加工、功能        |
| 关注重点 | 对象及对象之间的关系    | 输入、处理、输出         |
| 模型元素 | 类、对象、属性、方法、关系 | 外部实体、数据流、加工、数据存储 |
| 常用图  | 类图、用例图、顺序图    | DFD、数据字典         |
| 思维方式 | 以对象为中心        | 以功能 / 数据流为中心     |

### 试卷中的关键结论

面向对象分析的一个重要特点是：

```text
以问题域中的成分为基点来构造模型。
```

例如：

图书管理系统中，面向对象分析会关注：

```text
图书、借阅人、图书管理员、借阅记录、罚款单
```

结构化分析则关注：

```text
借书请求 → 查询库存 → 登记借阅 → 返回结果
```

### 易错点

| 说法                    | 正误 |
| --------------------- | -- |
| 面向对象分析使用抽象，结构化分析不使用抽象 | 错  |
| 两者都使用抽象               | 对  |
| 面向对象以问题域对象为中心         | 对  |
| 结构化分析以数据流和加工为中心       | 对  |

### 练习题

#### 题 1

面向对象分析与结构化分析的主要区别是什么？

**参考答案：**

面向对象分析以问题域中的对象、类和实体为中心建立模型；结构化分析以数据流和加工过程为中心建立模型。

---

#### 题 2

判断题：结构化分析方法不使用抽象。

**答案：×**

解析：结构化分析也使用抽象，例如抽象出数据流、加工和数据存储。

---

#### 题 3

在图书管理系统中，“图书”“借阅人”“借阅记录”更符合哪种分析方法的核心建模对象？

A. 结构化分析
B. 面向对象分析
C. 黑盒测试
D. 白盒测试

**答案：B**

---

## 6. 模块耦合

### 必须掌握

模块耦合是软件设计中的核心概念。

定义：

```text
模块耦合是指不同模块之间相互依赖、相互联系的程度。
```

设计目标：

```text
低耦合，高内聚
```

耦合越低，模块越独立，系统越容易维护、修改和复用。

### 常见耦合类型

| 耦合类型 | 含义                      | 例子                         | 好坏程度 |
| ---- | ----------------------- | -------------------------- | ---- |
| 内容耦合 | 一个模块直接访问或修改另一个模块内部数据或代码 | A 模块直接修改 B 模块内部变量          | 最差   |
| 公共耦合 | 多个模块共同访问全局数据            | 多个模块都读写 globalConfig       | 较差   |
| 外部耦合 | 多个模块依赖外部设备、协议或数据格式      | 多模块依赖同一外部文件格式              | 较强   |
| 控制耦合 | 一个模块传递控制信号影响另一个模块逻辑     | 参数 flag=1 表示打印，flag=2 表示保存 | 中等   |
| 标记耦合 | 模块之间传递复杂数据结构，但只使用其中一部分  | 传整个 Student 对象，只用 name 字段  | 中等   |
| 数据耦合 | 模块之间通过简单参数传递数据          | calculate(a, b)            | 较好   |

### 耦合强弱大致顺序

从差到好：

```text
内容耦合 → 公共耦合 → 外部耦合 → 控制耦合 → 标记耦合 → 数据耦合
```

### 试卷重点

试卷判断题考过：

```text
当一个模块直接修改另一个模块的数据时，这种耦合称为数据耦合。
```

答案：

```text
×
```

正确说法：

```text
这是内容耦合。
```

### 答题模板

如果简答题问“给出模块耦合定义，并举例说明 3 种模块耦合”，可以这样写：

```text
模块耦合是指软件系统中不同模块之间相互依赖的程度。耦合程度越低，模块独立性越强。

例如：
1. 内容耦合：一个模块直接访问或修改另一个模块的内部数据。
2. 公共耦合：多个模块共同访问同一个全局数据。
3. 数据耦合：模块之间仅通过简单参数传递数据。
```

### 练习题

#### 题 1

什么是模块耦合？

**参考答案：**

模块耦合是指软件系统中不同模块之间相互依赖、相互联系的程度。

---

#### 题 2

一个模块直接修改另一个模块的内部变量，这属于什么耦合？

**答案：内容耦合**

---

#### 题 3

模块 A 调用模块 B 时，只传递两个整数参数 `x` 和 `y`，模块 B 返回计算结果。这属于什么耦合？

**答案：数据耦合**

---

## 7. 模块内聚

### 必须掌握

虽然这份试卷主要考耦合，但内聚经常和耦合一起考。

定义：

```text
模块内聚是指一个模块内部各元素之间联系的紧密程度。
```

设计目标：

```text
高内聚，低耦合
```

内聚越高，说明模块职责越单一、越清晰。

### 常见内聚类型

从弱到强：

| 内聚类型 | 含义                 | 例子                    |
| ---- | ------------------ | --------------------- |
| 偶然内聚 | 模块内功能没有明显关系        | 一个模块里同时打印报表、计算工资、清空缓存 |
| 逻辑内聚 | 功能逻辑上类似，由参数决定执行哪一个 | 一个模块处理所有输入输出操作        |
| 时间内聚 | 功能在同一时间执行          | 初始化模块中打开文件、连接数据库、加载配置 |
| 过程内聚 | 按特定流程顺序执行          | 先读数据，再校验，再保存          |
| 通信内聚 | 多个操作使用同一数据         | 对同一个订单进行校验和计算总价       |
| 顺序内聚 | 前一部分输出作为后一部分输入     | 读取数据 → 解析数据 → 生成结果    |
| 功能内聚 | 只完成一个明确功能          | 只负责计算订单总价             |

### 易错点

| 说法            | 正误 |
| ------------- | -- |
| 偶然内聚最好        | 错  |
| 功能内聚最好        | 对  |
| 模块设计应追求高内聚低耦合 | 对  |
| 内聚越低越好        | 错  |

### 练习题

#### 题 1

软件设计中模块划分的理想原则是什么？

**参考答案：**

高内聚，低耦合。

---

#### 题 2

一个模块只完成“计算订单总价”这一项功能，属于什么内聚？

**答案：功能内聚**

---

#### 题 3

一个模块中同时包含“打印报表、清理日志、发送邮件”三个无关功能，属于什么内聚？

**答案：偶然内聚**

---

## 8. 黑盒测试与白盒测试

### 必须掌握

软件测试按是否关注程序内部结构，可以分为黑盒测试和白盒测试。

| 测试类型 | 是否看代码 | 测试依据     | 常见方法                 |
| ---- | ----- | -------- | -------------------- |
| 黑盒测试 | 不看代码  | 需求规格说明书  | 等价类划分、边界值分析、因果图、错误推测 |
| 白盒测试 | 看代码   | 程序内部逻辑结构 | 语句覆盖、判定覆盖、条件覆盖、路径覆盖  |

### 黑盒测试

黑盒测试把程序看成一个黑盒，只关心：

```text
输入是什么，输出是否正确。
```

常用方法：

1. 等价类划分；
2. 边界值分析；
3. 因果图；
4. 错误推测法。

### 白盒测试

白盒测试关注程序内部逻辑，例如：

1. 每条语句是否执行过；
2. 每个判断分支是否覆盖；
3. 每条路径是否测试过；
4. 条件真假是否都测试过。

## 10. N-S 图

### 必须掌握

N-S 图又称盒图，用来表示结构化程序的控制逻辑。

它只使用三种基本结构：

```text
顺序结构、选择结构、循环结构
```

### 三种基本结构

#### 1. 顺序结构

```text
S1
S2
S3
```

含义：

```text
先执行 S1，再执行 S2，再执行 S3。
```

#### 2. 选择结构

```text
IF 条件 THEN S1 ELSE S2
```

含义：

```text
条件为真执行 S1，否则执行 S2。
```

#### 3. 循环结构

```text
WHILE 条件 DO
    S
```

含义：

```text
只要条件为真，就重复执行 S。
```

### 本卷伪代码逻辑

试卷中有如下伪代码：

```text
BEGIN
S1;
S2;
IF a>5 THEN S3 ELSE S4;
n=1;
WHILE n<=5 DO
{
    S5;
    n=n+1;
}
IF b>5 THEN S6 ELSE S7;
END.
```

执行顺序是：

```text
开始
→ S1
→ S2
→ 判断 a>5
   → 是：S3
   → 否：S4
→ n=1
→ 判断 n<=5
   → 是：S5，n=n+1，然后继续判断
   → 否：退出循环
→ 判断 b>5
   → 是：S6
   → 否：S7
→ 结束
```

### 易错点

| 易错点           | 正确做法              |
| ------------- | ----------------- |
| 漏掉 n=1        | 必须画在循环之前          |
| 把 n=n+1 放到循环外 | 错，它属于循环体          |
| IF 分支没有画 ELSE | 要画出 S3 和 S4 两个分支  |
| WHILE 条件画成 IF | 循环要体现重复判断         |
| S6/S7 放到循环里面  | 错，第二个 IF 在循环结束后执行 |

### 练习题

#### 题 1

N-S 图主要用于描述什么？

**参考答案：**

用于描述结构化程序的控制逻辑，包括顺序、选择和循环结构。

---

#### 题 2

下面伪代码中，循环体包含哪些语句？

```text
i = 1;
WHILE i <= 10 DO
{
    S1;
    i = i + 1;
}
```

**答案：**

循环体包含：

```text
S1;
i = i + 1;
```

---

#### 题 3

判断题：N-S 图可以清晰表示顺序、选择、循环三种基本控制结构。

**答案：√**

## 13. DFD 数据流图

### 必须掌握

DFD 是数据流图，用于描述系统中数据如何流动、加工和存储。

### DFD 四个基本元素

| 元素   | 图形表示     | 含义             |
| ---- | -------- | -------------- |
| 外部实体 | 矩形       | 系统外部的人、组织或其他系统 |
| 加工   | 圆形或圆角矩形  | 对数据进行处理        |
| 数据流  | 箭头       | 数据的流动方向        |
| 数据存储 | 双横线或开口矩形 | 文件、数据库、数据表     |

### DFD 分层

| 层次           | 说明                           |
| ------------ | ---------------------------- |
| 顶层 DFD / 环境图 | 把整个系统看成一个加工，只显示系统与外部实体之间的数据流 |
| 0 层 DFD      | 将系统分解为若干主要加工                 |
| 1 层 DFD      | 对 0 层中的某个加工继续细化              |
| 底层 DFD       | 不能或不需要继续分解的详细图               |

### 顶层 DFD 画法

顶层 DFD 中：

1. 中间只有一个系统加工；
2. 周围是外部实体；
3. 外部实体和系统之间用数据流连接；
4. 数据流要有名称；
5. 一般不画系统内部细节。

### 图书管理系统中的外部实体

根据试卷描述，可以识别出：

| 外部实体  | 输入到系统的数据        | 系统输出给它的数据     |
| ----- | --------------- | ------------- |
| 借阅人   | 借书请求、还书信息、图书丢失单 | 无书通知、罚款单、借书结果 |
| 库房管理员 | 新书信息、图书卡片       | 购书通知          |
| 图书管理员 | 借还书处理信息、罚款处理信息  | 图书目录修改结果      |
| 图书馆主任 | 库存查询请求          | 月末库存量         |

也可以把“图书管理员”视为系统内部角色，具体要看建模边界。但考试答题时，关键是逻辑自洽。

### 易错点

| 易错点            | 正确做法         |
| -------------- | ------------ |
| 顶层 DFD 画很多内部模块 | 错，顶层只画一个系统加工 |
| 数据流没有命名        | 错，数据流必须命名    |
| 外部实体之间直接连线     | 错，数据流应通过系统   |
| 把数据库画成外部实体     | 错，数据库一般是数据存储 |
| 顶层 DFD 没有系统边界  | 错，应体现系统与外界交互 |

### 练习题

#### 题 1

DFD 的四个基本元素是什么？

**参考答案：**

外部实体、加工、数据流、数据存储。

---

#### 题 2

顶层 DFD 又称为什么？它主要表达什么？

**参考答案：**

顶层 DFD 又称系统环境图，主要表达系统与外部实体之间的数据流关系。

---

#### 题 3

在图书管理系统中，“借阅人”属于 DFD 的什么元素？

**答案：外部实体**

## 14. 数据字典

数据字典用于说明 DFD 中的数据流、数据存储、数据项的组成。

## 15. UML 顺序图

### 必须掌握

顺序图是 UML 中的一种交互图，用于描述对象之间消息传递的时间顺序。

### 基本元素

| 元素   | 含义            |
| ---- | ------------- |
| 参与者  | 系统外部的人或角色     |
| 对象   | 参与交互的系统对象     |
| 生命线  | 对象在交互过程中的存在时间 |
| 消息   | 对象之间发送的请求     |
| 返回消息 | 对请求的返回结果      |
| 激活条  | 对象正在执行操作的时间段  |
| 条件分支 | 不同条件下的不同交互路径  |

### 顺序图特点

```text
时间从上到下流动。
```

越靠上的消息越早发生，越靠下的消息越晚发生。

### 图书管理系统“借书”顺序图文字版

参与对象：

```text
借阅人、图书管理员、图书管理系统、图书目录文件
```

流程：

```text
1. 借阅人向图书管理员提出借书请求。
2. 图书管理员向图书管理系统提交借书信息。
3. 图书管理系统查询图书目录文件。
4. 图书目录文件返回库存状态。
5. 如果有库存：
   5.1 系统登记借阅信息。
   5.2 系统更新图书目录。
   5.3 图书管理员将图书交给借阅人。
6. 如果无库存：
   6.1 系统生成无书通知。
   6.2 图书管理员向借阅人发出无书通知。
```

### 图书管理系统“丢书处理”顺序图文字版

参与对象：

```text
借阅人、图书管理员、图书管理系统、图书目录文件、库房管理员
```

流程：

```text
1. 借阅人提交图书丢失单。
2. 图书管理员接收图书丢失单。
3. 图书管理员在系统中登记丢失信息。
4. 系统修改图书目录文件。
5. 系统生成罚款单。
6. 图书管理员将罚款单交给借阅人。
7. 系统通知库房管理员。
```

### 易错点

| 易错点         | 正确做法                |
| ----------- | ------------------- |
| 顺序图没有时间顺序   | 消息必须从上到下排列          |
| 参与对象太少      | 至少包含主要参与者和系统对象      |
| 没有返回结果      | 查询类操作应有返回消息         |
| 没有体现条件分支    | 有库存 / 无库存应分开描述      |
| 把 DFD 当成顺序图 | DFD 描述数据流，顺序图描述对象交互 |

### 练习题

#### 题 1

顺序图主要用于描述什么？

**参考答案：**

顺序图用于描述对象之间按照时间顺序发生的消息交互过程。

---

#### 题 2

顺序图中时间方向通常是怎样的？

**答案：从上到下**

#### 题 3

# 考前背诵版

最后给你一个最适合考前背的版本：

```text
1. 需求分析：定义需求并建立系统模型。
2. 总体设计：划分模块，得到优化的软件结构图。
3. 详细设计：设计模块算法和内部流程。
4. 测试计划：在需求分析阶段制定。
5. 结构化方法：自顶向下，逐步求精。
6. 面向对象分析：以问题域对象为中心。
7. 模块耦合：模块之间相互依赖程度。
8. 内容耦合：一个模块直接访问或修改另一个模块内部数据。
9. 数据耦合：模块之间通过简单参数传递数据。
10. 测试用例：应包含合理、不合理、边界、异常输入。
11. 边界值分析：黑盒测试。
12. 白盒测试：依据程序内部逻辑。
13. CMM 五级：初始级、可重复级、已定义级、已管理级、优化级。
14. CMM 2 级：需求管理、项目规划、项目跟踪、转包管理、质量保证、配置管理。
15. DFD 四元素：外部实体、加工、数据流、数据存储。
16. 顶层 DFD：系统环境图，只描述系统与外部实体的数据流。
17. 数据字典：用“数据流 = 数据项 + 数据项”的形式定义数据。
18. 顺序图：描述对象之间按时间顺序发生的消息交互。
```

这版如果你能全部背熟，并且能画出 N-S 图、PAD 图、顶层 DFD、顺序图，软件工程这部分基本就能冲满分。
