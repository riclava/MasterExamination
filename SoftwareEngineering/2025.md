## 2. 对象之间的交互：消息传递

### 必须掌握

面向对象方法中，对象之间不是直接访问彼此内部数据，而是通过 **消息传递** 实现协作。

例如：

```text
用户对象 → 发送“提交订单”消息 → 订单对象
订单对象 → 发送“扣减库存”消息 → 商品对象
```

### 常见考法

| 题型  | 考法                 |
| --- | ------------------ |
| 填空题 | 对象之间的交互通过____实现    |
| 选择题 | 对象间通信机制是什么         |
| 判断题 | 对象之间通过共享全局变量交互是否正确 |

### 易错点

| 容易混淆 | 正确答案       |
| ---- | ---------- |
| 函数调用 | 面向过程中的常见说法 |
| 数据流  | 结构化分析中的概念  |
| 消息传递 | 面向对象对象交互机制 |

### 练习题

**题 1：填空题**
面向对象方法中，对象之间的交互通过 ______ 实现。

**答案：**
消息传递。

## 4. 封装、信息隐蔽、最小界面、继承

### 必须掌握

这组概念是选择题高频考点。

| 概念   | 含义                  | 典型考法       |
| ---- | ------------------- | ---------- |
| 封装   | 把数据和操作绑定在一起，并隐藏内部实现 | 信息隐蔽靠什么实现  |
| 信息隐蔽 | 隐藏模块或对象内部细节，只暴露必要接口 | 软件设计原则     |
| 最小界面 | 公共接口只提供真正有用的操作      | 判断公共成员是否必要 |
| 继承   | 子类复用父类属性和方法         | 子类复用父类特征   |

### 必须会区分

| 题干关键词          | 答案   |
| -------------- | ---- |
| 隐藏对象内部实现       | 封装   |
| 只暴露必要接口        | 信息隐蔽 |
| 公共接口只保留用户需要的成员 | 最小界面 |
| 子类复用父类属性和方法    | 继承   |

### 易错点

| 容易选错         | 原因                |
| ------------ | ----------------- |
| 把信息隐蔽选成继承    | 继承是复用，不是隐藏        |
| 把最小界面选成信息隐蔽  | 最小界面更强调“接口少而必要”   |
| 把封装和信息隐蔽完全等同 | 封装是实现机制，信息隐蔽是设计原则 |

### 练习题

**题 1：选择题**

A. 分类性
B. 继承性

**答案：**
C。

**题 2：选择题**

## 5. 敏捷过程模型

### 必须掌握

敏捷过程模型的核心是：

| 内容   | 标准说法               |
| ---- | ------------------ |
| 开发方式 | 迭代增量               |
| 文档态度 | 消除不必要的文档，而不是完全不要文档 |
| 活动关系 | 分析、设计、编码、测试紧密结合    |
| 过程策略 | 反馈修正、反复测试、持续改进     |
| 交付特点 | 每个迭代都产出可运行的软件增量    |

### 必须会背

> 敏捷过程模型是一个迭代增量开发过程，消除了开发中不必要的文档和提交物，将分析、设计、编码和测试结合在一起，全过程采用迭代增量开发、反馈修正和反复测试的策略。

### 常见考法

| 题型  | 考法                |
| --- | ----------------- |
| 选择题 | 挖空考“迭代增量”“文档”“结合” |
| 判断题 | 敏捷是否完全不要文档        |
| 概念题 | 敏捷和瀑布模型区别         |

### 易错点

| 错误理解               | 正确理解          |
| ------------------ | ------------- |
| 敏捷就是不写文档           | 敏捷是不写无价值的过度文档 |
| 敏捷是线性开发            | 敏捷是迭代增量       |
| 敏捷中分析、设计、编码、测试严格分离 | 敏捷中这些活动紧密结合   |
| 敏捷一次性交付            | 敏捷持续交付增量      |

## 7. 模块耦合类型

### 必须掌握

模块耦合表示模块之间相互依赖的程度。耦合越低，模块独立性越好。

### 耦合类型从好到差

| 排序 | 耦合类型 | 含义               | 好坏 |
| -- | ---- | ---------------- | -- |
| 1  | 数据耦合 | 模块间通过参数传递简单数据    | 最好 |
| 2  | 控制耦合 | 模块间传递控制标志或控制信息   | 较差 |
| 3  | 公共耦合 | 多个模块共享全局数据或公共数据区 | 差  |
| 4  | 内容耦合 | 一个模块直接访问另一个模块内部  | 最差 |

### 记忆口诀

> 数据最好，内容最差；控制传标志，公共靠共享。

### 常见考法

| 题干关键词            | 答案   |
| ---------------- | ---- |
| 传递基本数据类型参数       | 数据耦合 |
| 传递标志位、开关量        | 控制耦合 |
| 多个模块共享全局变量       | 公共耦合 |
| 直接访问另一个模块内部代码或数据 | 内容耦合 |

### 易错点

| 错误理解          | 正确理解            |
| ------------- | --------------- |
| 数据耦合最差        | 数据耦合通常最好        |
| 内容耦合最好        | 内容耦合最差          |
| 传递控制标志也是数据耦合  | 传递控制标志是控制耦合     |
| 使用全局变量方便所以耦合低 | 全局共享属于公共耦合，耦合较高 |

### 练习题

**题 1：选择题**
结构化设计中，模块间通过参数传递基本数据类型的耦合属于：

A. 数据耦合
B. 控制耦合
C. 公共耦合
D. 内容耦合

**答案：**
A。

---

**题 2：选择题**
多个模块共同访问同一个全局变量，这种耦合属于：

A. 数据耦合
B. 控制耦合
C. 公共耦合
D. 内容耦合

**答案：**
C。

---

**题 3：判断题**

**答案：**
√。

## 8. 结构化分析与 DFD 基础

### 必须掌握

DFD 是 **Data Flow Diagram，数据流图**。

它描述：

| DFD 描述什么  | DFD 不描述什么        |
| --------- | ---------------- |
| 数据从哪里来    | 不描述程序执行顺序        |
| 数据到哪里去    | 不描述 if/else 控制逻辑 |
| 数据被哪个加工处理 | 不描述循环结构          |
| 数据存储在哪里   | 不描述具体算法步骤        |

### DFD 的四种基本元素

| 元素      | 图形习惯     | 含义           |
| ------- | -------- | ------------ |
| 外部实体    | 方框       | 系统外部的人、组织、系统 |
| 加工 / 处理 | 圆形或圆角框   | 对数据进行处理的功能   |
| 数据流     | 箭头       | 数据的流动方向      |
| 数据存储    | 双横线或开口矩形 | 文件、表、数据库     |

### 易错点

| 错误说法       | 正确说法         |
| ---------- | ------------ |
| DFD 描述控制流程 | DFD 描述数据流动   |
| DFD 等于流程图  | DFD 不是流程图    |
| DFD 要画判断条件 | DFD 不重点画控制判断 |
| 数据流可以没有名字  | 数据流通常要命名     |

### 练习题

**题 1：判断题**
结构化分析中，数据流图用于描述系统的控制流程。

**答案：**
×。数据流图描述数据流动和处理过程，不是控制流程。

---

**题 2：选择题**
DFD 中，用来表示系统外部参与者的是：

A. 数据流
B. 数据存储
C. 外部实体
D. 加工

**答案：**
C。

---

**题 3：选择题**
DFD 中，用箭头表示的是：

A. 加工
B. 数据流

D. 数据存储

**答案：**

## 9. 结构化开发方法原则

### 必须掌握

结构化开发方法的核心原则是：

> 自顶向下、逐步求精。

### 什么叫自顶向下？

先看系统整体，再分解成子系统、模块、功能。

```text
销售管理系统
 ├── 请求校验
 │    ├── 用户信誉校验
 │    └── 商品库存校验
 └── 订单处理
```

### 什么叫逐步求精？

每一层都比上一层更细，但不能一开始就陷入具体代码。

### 常见考法

| 题型  | 考法                         |
| --- | -------------------------- |
| 判断题 | 结构化方法是否自顶向下                |
| 填空题 | 自顶向下、逐步求精                  |
| 建模题 | 顶层 DFD → 0 层 DFD → 1 层 DFD |

### 易错点

| 错误做法       | 正确做法        |
| ---------- | ----------- |
| 一开始就画最细模块  | 先画顶层，再逐层分解  |
| 分解后输入输出不一致 | 分解后要保持数据平衡  |
| 把控制流程当数据流  | DFD 只关注数据流动 |

### 练习题

**题 1：填空题**
结构化开发方法的核心指导原则是 ******、******。

**答案：**
自顶向下；逐步求精。

---

**题 2：判断题**
结构化分析通常从系统整体开始，逐层分解为更小的功能或加工。

**答案：**
√。

---

**题 3：选择题**
下列最符合结构化方法思想的是：

A. 先写代码，再补需求
B. 自底向上随意拼接模块
C. 自顶向下、逐步求精
D. 只关注界面效果

**答案：**
C。

## 10. 软件测试基本原则

### 必须掌握

软件测试的核心结论：

| 观点                   | 正误 |
| -------------------- | -- |
| 测试的目的是发现错误           | 对  |
| 测试可以证明软件没有错误         | 错  |
| 没发现错误不代表没有错误         | 对  |
| 测试无法穷尽所有情况           | 对  |
| 测试只能证明缺陷存在，不能证明缺陷不存在 | 对  |

### 常见考法

| 题型  | 考法                  |
| --- | ------------------- |
| 判断题 | “测试没有发现问题，说明软件没有错误” |
| 选择题 | 软件测试的目的             |
| 概念题 | 测试的局限性              |

### 易错点

| 错误理解        | 正确理解             |
| ----------- | ---------------- |
| 测试通过 = 软件无错 | 测试通过只代表当前测试未发现错误 |
| 测试是为了证明程序正确 | 测试主要是为了发现错误      |
| 测试可以覆盖所有情况  | 实际上通常无法穷尽所有输入和路径 |

### 练习题

| 内容   | 例子           |
| ---- | ------------ |
| 命名规范 | 变量名、函数名清晰    |
| 代码格式 | 缩进一致、结构清楚    |
| 注释习惯 | 必要位置添加注释     |
| 逻辑表达 | 避免复杂、冗余、重复逻辑 |
| 模块组织 | 函数职责单一、代码易维护 |

### 本卷考点

> 编程风格与程序运行效率相关。

这个说法在试卷中判断为正确。理解时要注意：不是说“缩进本身会提升运行效率”，而是良好的编程风格可能减少冗余逻辑、提升代码结构质量，从而间接影响效率。

### 易错点

| 错误理解           | 正确理解            |
| -------------- | --------------- |
| 编程风格只影响美观      | 还影响可读性、可维护性、可靠性 |
| 编程风格一定直接提升运行速度 | 多数情况下是间接影响      |
| 注释越多越好         | 注释应必要、准确、不过度    |

### 练习题

编程风格与程序的运行效率完全无关。

**答案：**

# 12. DFD 建模题：顶层 DFD、0 层 DFD、1 层 DFD

这是本卷最重要的大题。试卷要求针对销售管理系统绘制：顶层 DFD、0 层 DFD 和 1 层 DFD。题干中给出的外部实体有用户、供应商、销售经理；数据存储有用户信息表、商品表、订单文件；核心加工包括请求校验、订单处理；请求校验进一步分解为用户信誉校验和商品库存校验。

---

## 12.1 DFD 识别方法

拿到题干后，先做四类标记。

### 第一步：找外部实体

题干中出现的 **系统外部的人、组织、外部系统**，通常是外部实体。

| 题干词  | DFD 元素 |
| ---- | ------ |
| 用户   | 外部实体   |
| 供应商  | 外部实体   |
| 销售经理 | 外部实体   |

### 第二步：找数据存储

题干中出现的 **表、文件、库、记录、档案**，通常是数据存储。

| 题干词   | DFD 元素 |
| ----- | ------ |
| 用户信息表 | 数据存储   |
| 商品表   | 数据存储   |
| 订单文件  | 数据存储   |

### 第三步：找加工

题干中表示 **处理、校验、生成、统计、查询、更新** 的动词短语，通常是加工。

| 题干词    | DFD 元素                  |
| ------ | ----------------------- |
| 请求校验   | 加工                      |
| 订单处理   | 加工                      |
| 用户信誉校验 | 加工                      |
| 商品库存校验 | 加工                      |
| 生成销售报表 | 可作为订单处理的一部分，或单独加工，视题目要求 |

### 第四步：找数据流

题干中表示 **请求、信息、结果、通知、报表、数据** 的名词短语，通常是数据流。

| 题干词    | DFD 元素 |
| ------ | ------ |
| 商品购买请求 | 数据流    |
| 用户信息   | 数据流    |
| 商品库存   | 数据流    |
| 校验通过   | 数据流    |
| 订单信息   | 数据流    |
| 缺货通知   | 数据流    |
| 缺货信息   | 数据流    |
| 库存更新   | 数据流    |
| 销售报表   | 数据流    |

---

## 12.2 顶层 DFD 怎么画

### 顶层 DFD 的特点

顶层 DFD 又叫 **系统环境图**。

它只画：

```text
外部实体 → 系统 → 外部实体
```

顶层图一般不展开系统内部细节。

### 本题顶层 DFD 应包含

| 外部实体 | 输入到系统  | 系统输出        |
| ---- | ------ | ----------- |
| 用户   | 商品购买请求 | 订单信息 / 缺货通知 |
| 供应商  | 库存更新   | 缺货信息        |
| 销售经理 | 查询订单请求 | 销售报表        |

### 顶层 DFD 文字版

```text
用户 ──商品购买请求──> 销售管理系统
销售管理系统 ──订单信息 / 缺货通知──> 用户

销售管理系统 ──缺货信息──> 供应商
供应商 ──库存更新──> 销售管理系统

销售经理 ──查询订单请求──> 销售管理系统
销售管理系统 ──销售报表──> 销售经理
```

### 顶层 DFD 满分要点

| 要点         | 是否必须 |
| ---------- | ---- |
| 系统作为一个整体加工 | 必须   |
| 外部实体完整     | 必须   |
| 输入输出数据流完整  | 必须   |
| 不画系统内部加工   | 通常不画 |
| 不画代码逻辑     | 不能画  |

---

## 12.3 0 层 DFD 怎么画

### 0 层 DFD 的特点

0 层 DFD 是把顶层 DFD 里的“销售管理系统”拆成几个核心加工。

本题明确要求拆成：

```text
1 请求校验
2 订单处理
```

### 0 层 DFD 应包含的加工

| 编号 | 加工名称 | 作用                      |
| -- | ---- | ----------------------- |
| 1  | 请求校验 | 校验用户信誉和商品库存             |
| 2  | 订单处理 | 生成订单、写入订单文件、返回订单结果、生成报表 |

### 0 层 DFD 应包含的数据存储

| 数据存储  | 被哪个加工使用      | 数据流    |
| ----- | ------------ | ------ |
| 用户信息表 | 请求校验         | 用户信息   |
| 商品表   | 请求校验         | 商品库存   |
| 商品表   | 请求校验 / 供应商更新 | 库存更新   |
| 订单文件  | 订单处理         | 订单信息   |
| 订单文件  | 销售经理查询       | 查询订单信息 |

### 0 层 DFD 文字版

```text
用户 ──商品购买请求──> 1 请求校验

用户信息表 ──用户信息──> 1 请求校验
商品表 ──商品库存──> 1 请求校验

1 请求校验 ──校验通过──> 2 订单处理
1 请求校验 ──缺货通知──> 用户
1 请求校验 ──缺货信息──> 供应商

供应商 ──库存更新──> 商品表

2 订单处理 ──订单信息──> 订单文件
2 订单处理 ──订单信息──> 用户

销售经理 ──查询订单信息──> 订单文件
订单文件 ──订单信息──> 2 订单处理
2 订单处理 ──生成销售报表──> 销售经理
```

### 0 层 DFD 满分要点

| 检查项               | 说明  |
| ----------------- | --- |
| 是否有 1 请求校验        | 必须有 |
| 是否有 2 订单处理        | 必须有 |
| 用户购买请求是否进入请求校验    | 必须  |
| 请求校验是否访问用户信息表、商品表 | 必须  |
| 校验通过是否流向订单处理      | 必须  |
| 订单处理是否写订单文件       | 必须  |
| 缺货时是否通知用户和供应商     | 必须  |
| 销售经理是否能获得销售报表     | 必须  |

---

## 12.4 1 层 DFD 怎么画

### 1 层 DFD 的特点

1 层 DFD 是继续分解某一个加工。

本题要求把：

```text
1 请求校验
```

分解为：

```text
1.1 用户信誉校验
1.2 商品库存校验
```

### 两个加工的职责

| 编号  | 加工名称   | 输入          | 输出                   |
| --- | ------ | ----------- | -------------------- |
| 1.1 | 用户信誉校验 | 商品购买请求、用户信息 | 用户校验通过 / 用户校验失败      |
| 1.2 | 商品库存校验 | 用户校验通过、商品库存 | 库存校验通过 / 缺货通知 / 缺货信息 |

### 1 层 DFD 文字版

```text
用户 ──商品购买请求──> 1.1 用户信誉校验

用户信息表 ──用户信息──> 1.1 用户信誉校验

1.1 用户信誉校验 ──用户校验通过──> 1.2 商品库存校验
1.1 用户信誉校验 ──用户校验失败 / 拒绝信息──> 用户

商品表 ──商品库存──> 1.2 商品库存校验

1.2 商品库存校验 ──库存校验通过──> 2 订单处理
1.2 商品库存校验 ──缺货通知──> 用户
1.2 商品库存校验 ──缺货信息──> 供应商

供应商 ──库存更新──> 商品表
```

### 1 层 DFD 满分要点

| 检查项                   | 说明  |
| --------------------- | --- |
| 是否把“请求校验”拆成 1.1 和 1.2 | 必须  |
| 1.1 是否访问用户信息表         | 必须  |
| 1.2 是否访问商品表           | 必须  |
| 1.1 到 1.2 是否有“用户校验通过” | 建议有 |
| 1.2 到订单处理是否有“库存校验通过”  | 必须  |
| 缺货是否通知用户和供应商          | 必须  |
| 数据流名称是否清楚             | 必须  |

---

## 12.5 DFD 建模常见扣分点

| 扣分点                  | 为什么错                    |
| -------------------- | ----------------------- |
| 把用户信息表画成外部实体         | 表是数据存储，不是外部实体           |
| 把供应商画成数据存储           | 供应商是外部实体                |
| 数据流没有名字              | DFD 数据流必须表达流动的数据内容      |
| 箭头方向反了               | 数据流方向错误会影响语义            |
| 顶层 DFD 画太细           | 顶层图只展示系统边界              |
| 0 层 DFD 没有数据存储       | 缺少关键数据来源                |
| 1 层 DFD 与 0 层输入输出不平衡 | 违反数据平衡原则                |
| 把控制条件画成数据流           | DFD 关注数据，不是控制流程         |
| 加工名称用名词              | 加工最好用动宾结构，如“校验请求”“处理订单” |
| 数据存储直接连外部实体          | 一般应通过加工访问数据存储           |

# 最后给你一版“满分复习清单”

## 第一优先级：必须拿满

| 判断句             | 正误 |
| --------------- | -- |
| DFD 描述系统控制流程    | ×  |
