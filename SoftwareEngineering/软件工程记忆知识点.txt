【软件工程】【总览】软件工程高频模块包括生命周期与需求设计、软件测试、结构化分析与 DFD、面向对象、软件设计、过程模型、过程改进与管理、补充考点。
【软件工程】【生命周期】可行性研究回答“值不值得做”，主要分析技术、经济、操作、法律可行性，产物是可行性研究报告。
【软件工程】【生命周期】需求分析回答“软件要做什么”，主要获取、分析、定义需求并建立逻辑模型，产物是 SRS、用例图、DFD、ER 图。
【软件工程】【生命周期】概要设计回答“系统总体怎么分”，主要划分模块、确定体系结构和模块接口，产物是软件结构图或模块结构图。
【软件工程】【生命周期】详细设计回答“模块内部怎么实现”，主要设计算法、数据结构和控制流程，产物是详细设计说明书、N-S 图、PAD 图。
【软件工程】【生命周期】编码阶段把设计结果用程序语言实现，主要产物是源代码。
【软件工程】【生命周期】测试阶段通过执行测试用例发现错误并验证需求是否满足，产物是测试报告和缺陷报告。
【软件工程】【生命周期】维护阶段修改已交付软件，包括纠错、适应、完善、预防，产物是新版本和维护记录。
【软件工程】【生命周期】需求分析关注“做什么”，不是“怎么做”。
【软件工程】【生命周期】概要设计确定总体结构和模块关系。
【软件工程】【生命周期】详细设计确定模块算法和内部流程。
【软件工程】【生命周期】测试计划通常在需求分析阶段制定。
【软件工程】【生命周期】看到“用户需求、功能需求、非功能需求、逻辑模型、SRS”，通常对应需求分析。
【软件工程】【生命周期】看到“总体结构、模块划分、模块接口、体系结构”，通常对应概要设计。
【软件工程】【生命周期】看到“模块算法、数据结构、处理流程、N-S 图、PAD 图”，通常对应详细设计。
【软件工程】【生命周期】看到“源程序、程序语言实现”，通常对应编码。
【软件工程】【生命周期】看到“发现错误、验证需求、执行测试用例”，通常对应测试。

【软件工程】【需求分析】需求分析的主要任务是理解用户需求、定义功能需求和非功能需求、建立系统逻辑模型、编写需求规格说明书。
【软件工程】【需求分析】功能需求描述系统必须提供的功能，例如登录、查询、支付、生成报表。
【软件工程】【需求分析】非功能需求描述性能、可靠性、安全性等质量要求，例如 2 秒内响应、支持 1000 人并发。
【软件工程】【需求分析】约束条件描述开发或运行必须满足的限制，例如必须使用某数据库、必须兼容某浏览器。
【软件工程】【需求分析】需求分析阶段的答题模板是：定义软件需求并建立系统逻辑模型，关注软件“做什么”，主要产物是需求规格说明书和相关需求模型。
【软件工程】【需求分析】测试目标、测试范围和通过标准都应依据需求规格说明。
【软件工程】【概要设计】概要设计也称总体设计。
【软件工程】【概要设计】软件结构图通常是概要设计阶段的产物。
【软件工程】【详细设计】N-S 图和 PAD 图常用于详细设计阶段。
【软件工程】【详细设计】模块内部算法属于详细设计，不属于概要设计。
【软件工程】【过程模型】瀑布模型的典型流程是需求分析 → 概要设计 → 详细设计 → 编码 → 测试 → 运行维护。
【软件工程】【过程模型】瀑布模型阶段划分清楚，每个阶段有明确输入输出，前一阶段完成后再进入后一阶段。
【软件工程】【过程模型】瀑布模型适合需求明确稳定、变化少的项目。
【软件工程】【过程模型】瀑布模型不适合需求频繁变化的大型复杂项目。
【软件工程】【过程模型】增量模型把系统分成若干功能增量，每次开发并交付一部分功能。
【软件工程】【过程模型】演化模型通过多次迭代和用户反馈，让软件从初始版本逐步演化为最终系统。
【软件工程】【过程模型】增量模型重点是分批交付功能。
【软件工程】【过程模型】演化模型重点是根据反馈不断修改完善。
【软件工程】【过程模型】螺旋模型将瀑布模型和演化模型结合起来，并增加风险分析。
【软件工程】【过程模型】螺旋模型以原型为基础，每进行一轮螺旋式开发就得到软件的一个新版本。
【软件工程】【过程模型】螺旋模型适合大型、复杂、高风险的软件项目。
【软件工程】【过程模型】螺旋模型的四个典型活动是制定计划、风险分析、工程实现、客户评估。
【软件工程】【过程模型】螺旋模型的核心特点是风险驱动。
【软件工程】【过程模型】喷泉模型是面向对象开发模型，体现迭代和无间隙特征。
【软件工程】【过程模型】喷泉模型不严格区分分析、设计和实现阶段，允许活动交叉和反复进行。
【软件工程】【过程模型】喷泉模型关键词是面向对象、迭代、无间隙、不严格线性。
【软件工程】【过程模型】敏捷开发强调迭代增量交付、快速响应需求变化、用户持续参与、团队协作。
【软件工程】【过程模型】敏捷开发认为可工作的软件比过度文档更重要。
【软件工程】【过程模型】敏捷不是不要文档，而是避免过度文档。
【软件工程】【过程模型】敏捷适合需求变化快、反馈频繁、可迭代交付的项目。
【软件工程】【过程模型】RUP 是 Rational Unified Process，即统一软件开发过程，是迭代式、增量式的软件过程。
【软件工程】【RUP】RUP 用“角色”回答谁做。
【软件工程】【RUP】RUP 用“活动/任务”回答做什么。
【软件工程】【RUP】RUP 用“制品”回答产生什么。
【软件工程】【RUP】RUP 用“工作流”回答怎么组织。
【软件工程】【RUP】RUP 的 6 个核心工程工作流是业务建模、需求、分析与设计、实现、测试、部署。
【软件工程】【RUP】RUP 的 3 个核心支持工作流是项目管理、配置与变更管理、环境。
【软件工程】【RUP】RUP 中系统分析员、设计员、测试员属于角色。
【软件工程】【RUP】RUP 中用例模型、设计模型、测试报告属于制品。
【软件工程】【RUP】RUP 中负责安装、配置、用户培训和软件交付的是部署工作流。
【软件工程】【RAD】RAD 是 Rapid Application Development，即快速应用程序开发。
【软件工程】【RAD】RAD 包括业务建模、数据建模和流程建模。
【软件工程】【RAD】业务建模分析业务流程和信息流。
【软件工程】【RAD】数据建模识别数据对象、属性和关系。
【软件工程】【RAD】流程建模描述数据如何被增加、修改、删除、查询和加工。
【软件工程】【面向对象】类是对具有相同属性和行为的一组对象的抽象描述。
【软件工程】【面向对象】对象是类的具体实例。
【软件工程】【面向对象】属性是对象具有的数据特征。
【软件工程】【面向对象】方法或操作是对象能够执行的行为。
【软件工程】【面向对象】类是对象的抽象，对象是类的实例。
【软件工程】【面向对象】对象由属性和操作组成，不是只包含数据。
【软件工程】【面向对象】封装是把数据和操作绑定起来，并隐藏内部实现，通过接口访问对象。
【软件工程】【面向对象】继承是子类复用父类的属性和方法。
【软件工程】【面向对象】多态是同一消息作用于不同对象，产生不同响应。
【软件工程】【面向对象】重载是在同一类中方法名相同但参数列表不同，通常编译期决定。
【软件工程】【面向对象】重写是子类重新定义父类已有方法，通常运行时绑定。
【软件工程】【面向对象】看到“隐藏对象内部实现”，选封装。
【软件工程】【面向对象】看到“子类复用父类”，选继承。
【软件工程】【面向对象】看到“同一消息，不同对象，不同响应”，选多态。
【软件工程】【面向对象】看到“同名方法，参数列表不同”，选重载。
【软件工程】【面向对象】看到“子类重新定义父类方法”，选重写。
【软件工程】【面向对象】对象之间通过消息传递进行协作。
【软件工程】【面向对象】消息传递中发送者只需知道接收对象接口，不必知道内部实现。
【软件工程】【面向对象】消息传递支持封装、多态和动态绑定。
【软件工程】【面向对象】传统过程调用是对函数或过程的直接调用，调用关系相对固定。
【软件工程】【面向对象】接收消息的对象可以执行相应操作，但不一定必须返回回答。
【软件工程】【面向对象】对象生命周期可概括为创建 → 使用/等待 → 不再引用 → 回收。
【软件工程】【面向对象】对象状态表示对象在某一时刻的条件或情况，状态会因事件或消息而改变。
【软件工程】【面向对象分析】对象模型描述系统静态结构，回答“系统中有什么”，常用类图、对象图。
【软件工程】【面向对象分析】动态模型描述系统动态行为，回答“系统怎样运行、对象如何变化”，常用状态图、顺序图、活动图。
【软件工程】【面向对象分析】功能模型描述系统功能与数据处理，回答“系统要做什么”，常用用例图、DFD。
【软件工程】【4+1 视图】逻辑视图描述类、对象和功能关系，记忆关键词是功能。
【软件工程】【4+1 视图】开发视图描述模块、包和代码组织，记忆关键词是代码。
【软件工程】【4+1 视图】物理视图描述服务器、网络节点和部署结构，记忆关键词是部署。
【软件工程】【4+1 视图】进程视图描述进程、线程、并发和通信，记忆关键词是运行。
【软件工程】【4+1 视图】场景视图或用例视图用用例串联和验证其他视图，记忆关键词是用例。
【软件工程】【4+1 视图】4+1 视图口诀是逻辑看功能，开发看代码，物理看部署，进程看运行，场景看用例。
【软件工程】【UML 关系】依赖表示一个类临时使用另一个类，例如 A 的方法参数使用 B。
【软件工程】【UML 关系】关联表示两个类存在结构性联系，例如学生选课。
【软件工程】【UML 关系】聚合表示弱整体/部分关系，部分可独立存在，例如班级和学生。
【软件工程】【UML 关系】组合表示强整体/部分关系，部分生命周期依赖整体，例如订单和订单明细。
【软件工程】【UML 关系】泛化表示继承关系，例如学生是人。
【软件工程】【UML 关系】实现表示类实现接口，例如类实现 Runnable。
【软件工程】【UML 关系】整体/部分关系通常用聚合或组合，不用依赖。
【软件工程】【UML 关系】组合比聚合更强，部分对象通常不能脱离整体单独存在。
【软件工程】【状态图】状态图用于描述对象或系统在事件驱动下的状态变化。
【软件工程】【状态图】状态图核心元素包括初始状态、状态、转移、事件、条件/守卫条件、动作、终止状态。
【软件工程】【状态图】状态图转移常写作“状态A --事件[条件]/动作--> 状态B”。
【软件工程】【状态图】状态图建模先找状态，再找事件，再找条件，再找异常路径，最后补全状态名。
【软件工程】【状态图】状态图描述动态状态变化，不描述静态类结构。
【软件工程】【顺序图】顺序图用于描述对象之间消息传递的时间顺序。
【软件工程】【顺序图】顺序图基本元素包括参与者、对象、生命线、消息、返回消息、激活条。
【软件工程】【顺序图】顺序图中的时间方向通常是从上到下。
【软件工程】【顺序图】顺序图答题可按“参与对象包括……首先……然后……若条件成立……否则……”组织。
【软件工程】【软件设计】内聚表示模块内部各部分联系的紧密程度，内聚越高越好。
【软件工程】【软件设计】耦合表示模块之间相互依赖的程度，耦合越低越好。
【软件工程】【软件设计】优秀模块设计追求高内聚、低耦合。
【软件工程】【内聚】内聚从低到高排序是偶然 < 逻辑 < 时间 < 过程 < 通信 < 顺序 < 功能。
【软件工程】【内聚】偶然内聚是模块内部功能毫无关系，例如把无关工具函数放一起。
【软件工程】【内聚】逻辑内聚是逻辑上相似的功能通过控制参数选择执行，例如 opType 决定增删改查。
【软件工程】【内聚】时间内聚是因为同一时间执行而放在一起，例如初始化时打开文件和读取配置。
【软件工程】【内聚】过程内聚是按固定步骤执行但数据联系不强，例如先验证再打印再记录。
【软件工程】【内聚】通信内聚是多个操作使用同一输入或输出数据，例如都处理同一客户资料。
【软件工程】【内聚】顺序内聚是前一部分输出作为后一部分输入，例如读取订单、计算金额、生成发票。
【软件工程】【内聚】功能内聚是模块只完成一个明确功能，例如计算订单总价。
【软件工程】【内聚】通过控制参数决定执行哪个功能，属于逻辑内聚。
【软件工程】【内聚】前一步输出作为后一步输入，属于顺序内聚。
【软件工程】【内聚】所有操作围绕一个明确功能，属于功能内聚。
【软件工程】【内聚】多个操作使用同一数据，属于通信内聚。
【软件工程】【耦合】耦合从低到高常见排序是非直接耦合 < 数据耦合 < 标记耦合 < 控制耦合 < 外部耦合 < 公共耦合 < 内容耦合。
【软件工程】【耦合】非直接耦合是两个模块无直接关系。
【软件工程】【耦合】数据耦合是通过简单参数传递数据，属于较好的耦合。
【软件工程】【耦合】标记耦合是模块之间传递复合数据结构。
【软件工程】【耦合】控制耦合是传递控制参数影响对方逻辑。
【软件工程】【耦合】外部耦合是多个模块共同依赖外部格式、设备或协议。
【软件工程】【耦合】公共耦合是多个模块访问同一全局数据。
【软件工程】【耦合】内容耦合是一个模块直接访问或修改另一个模块内部，属于最差耦合。
【软件工程】【耦合】只传递必要简单数据，属于数据耦合。
【软件工程】【耦合】传递标志变量控制对方分支，属于控制耦合。
【软件工程】【耦合】多个模块共用全局变量，属于公共耦合。
【软件工程】【耦合】一个模块直接访问另一个模块内部数据，属于内容耦合。
【软件工程】【设计模式】设计模式是针对反复出现的软件设计问题总结出的可复用解决方案。
【软件工程】【设计模式】设计模式的价值是复用成熟设计经验、提高可维护性、提高可扩展性、降低设计复杂度。

【软件工程】【设计模式】单例模式、工厂方法模式、观察者模式都属于设计模式。
